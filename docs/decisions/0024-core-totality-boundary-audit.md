# Decision 0024: Core Totality Boundary Audit

## Status

Confirmed for the current `transparent-v0` reference engine slice.

## Context

The reference engine now executes `Function`, closure creation, Arrow
Copy/Drop, `Apply`, independent function instances, `NatRec`, higher-order
`NatRec`, and `ProgramPackage` entry execution. The project direction is a
System T-inspired total Core, but the implementation had not yet separated
what is structurally enforced from what is only intended.

The audit looked for accepted graphs that could be well-typed but unable to
progress, recursive template/capture shapes, self-application through Arrow
values, NatRec control-state escapes, and closure literal injection through
the package boundary.

## Decision

Validated Core graphs reject directed value dependency cycles. The cycle check
runs after node, port, direction, type, and connection-count validation and
returns a deterministic typed validation error:

```text
Cyclic_value_dependency(path)
```

This rejects self-cycles such as `Succ.result -> Succ.input` and indirect
cycles such as `A.result -> B.input` and `B.result -> A.input`. `NatRec` remains
the only implemented recursive control primitive in `transparent-v0`; it is
represented by one executable node with explicit bounded lifecycle state, not
by a directed graph cycle.

`ProgramPackage` also rejects closure payloads as package literals with a typed
validation error:

```text
Unsupported_program_literal_payload
```

Current package literals are materializable `Unit` and `Nat` values. Closure
values must be created by Core `Function` rewrites so their template identity,
capture order, logical ID, and provenance are generated by the reference
engine.

## Rationale

Directed value cycles can be type-correct at the port level while having no
runtime value origin. Before this decision, such graphs could validate and then
reach `Run_stuck`. Rejecting value cycles strengthens the progress boundary
without adding scheduler heuristics or hidden values.

Rejecting closure literals at the package boundary prevents callers from
injecting closure payloads whose template identity, captured values, or
provenance were not created by validated Core execution. This keeps closure
environment construction acyclic under the current public API.

## Current Argument Level

This decision does not provide a machine-checked totality proof. The current
claim is:

- the implementation structurally enforces finite validated graphs,
  single-assignment ports, typed ports, instance-local node lifecycle,
  acyclic value dependency, explicit template dependency cycle rejection, and
  bounded `NatRec` lifecycle state;
- regression tests cover the discovered cycle and closure-literal boundaries,
  representative preservation checks through successful rewrites, waiting
  caller states, higher-order closure accumulators, and deterministic replay;
- a formal progress, preservation, and strong-normalization proof remains
  future work.

## Deferred

This decision does not implement:

- a proof assistant formalization,
- a complete termination checker,
- recursive types,
- general recursion,
- effectful Core,
- reachability policy beyond existing port connectivity and executable
  completion rules,
- canonical runtime-state serialization for automated state equivalence,
- `linear-v0` NatRec or totality analysis.
